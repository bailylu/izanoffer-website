# 0001 · C3 双栏交互模型

- **日期**: 2026-06-25
- **状态**: 已采纳
- **决策人**: Joe / 共同

---

## 背景

产品的核心 UI 形态需要拍板。过去两周尝试了多种布局：

- **A** · 侧边对话面板（官网常规展示 + 右侧 chat 面板常驻）
- **B** · 落地页 + 全屏对话（点击进入沉浸式聊天，类似 Claude）
- **C** · 双栏：官网原文 + AI 对话
- **C2** · 双栏 + 段落级双语原文（左边交错显示英文 + 中文）
- **C3** · 双栏 + 纯英文原文 + AI 完全在右边

产品定位也同步演化：从 v0.1 的「AI 替你做留学」收敛到 v0.3 的「AI 帮你看懂留学」——AI 是顾问 + 翻译者，不是代理人。这个定位变化要求 UI 必须能清晰区分「事实」（官网原文）和「判断」（AI 解读）。

## 候选方案

### 方案 A · 侧边对话面板
- 描述：官网营销页常规布局，右侧持续展示一个 ~420px 对话面板
- 优点：用户随时能开聊，"AI 顾问"存在感持续
- 缺点：营销区被挤窄；对话区也窄；移动端基本无法用；看起来像"网站有 AI 插件"

### 方案 B · 全屏对话（落地页 → 全屏）
- 描述：首页有 Hero 输入框，提交后过渡到全屏对话（Claude 模式）
- 优点：首页能好好讲故事；对话进入沉浸态；像真正的产品
- 缺点：AI 自说自话，没有信任锚；对高决策成本的留学场景信任度不够

### 方案 C · 双栏（官网 + AI）
- 描述：左边官网内容 + 右边 AI 对话；引用反向高亮
- 优点：官网原文在场建立信任锚；事实和判断有空间分开
- 缺点：左边内容由 AI 整理过，学生疑问"AI 是否动过手脚"

### 方案 C2 · 双栏 + 段落级双语
- 描述：左边原文 + 翻译段落交错显示；提供"双语/原文/中文"三种模式切换
- 优点：双语并存，原文可信
- 缺点：双语段落交错增加阅读复杂度；用户需要主动切换模式，决策成本高

### 方案 C3 · 双栏 + 纯原文 + AI 全包
- 描述：左边只有官网原文（不修改、不嵌入翻译）；右边 AI 全部解读（翻译 + 总结 + 建议）；划词即解读；URL 可粘贴
- 优点：事实 / 判断物理隔离，心智最清晰；零模式切换；AI 价值在对话框里释放完毕
- 缺点：左边只有英文，对完全不懂英文的家长不够友好（但可以通过划词解决）

## 决定

我们选了 **方案 C3**。

## 理由

1. **事实和判断的物理隔离**是高决策成本场景的最佳做法。学生看左边知道"官网说什么"，看右边知道"AI 怎么解读"——心智清晰，没有"AI 改过原文吗"的疑虑。

2. **零模式切换** = 最简产品。C2 的"双语/原文/中文"切换会让用户每次进来都要做一个无意义的决策。C3 让用户什么都不选，体验最丝滑。

3. **AI 价值在对话框里释放更彻底**。在 C/C2 里 AI 跨左右两边，能做的有限。C3 把 AI 集中到右边，可以做长翻译、追问、对照学生背景、引用案例库——表达空间更大。

4. **划词即解读** + **URL 粘贴** 两个交互让 C3 既保持简单，又有任意扩展性。学生想问任何一段都能选中让 AI 解读；想看任何 URL 都能粘贴进来。覆盖了 C 和 C2 想解决但解决得复杂的所有场景。

5. **跟 Claude / ChatGPT 用户心智一致**。"文档在左，AI 在右"是用户已经熟悉的范式，零学习成本。

## 后果

**正面**：
- 产品形态可以一句话讲清楚（"左边官网，右边 AI"），融资 / 销售 / 招人都好讲
- 工程实现最简单：左栏只渲染快照，右栏标准 chat UI
- 信任感最强：原文未经任何 AI 加工
- AI 政策合规：我们不改原文、不写文书，只解读

**负面 / 取舍**：
- 完全不懂英文的家长用户体验稍弱（缓解：划词翻译 + 顶部"翻译整页"按钮可作为 v0.4 选项）
- 左栏看起来"静态"（缓解：右上角"↗ 在原网站打开"提供动作出口）

**需要立刻做的事**：
- [ ] 设计 C3 在签证 / 案例库 / 表单等场景下的具体形态（Pattern A/B/C/D 四种左侧形态）
- [ ] 工程实现左栏的官网快照渲染 pipeline
- [ ] 实现划词浮动按钮 + 选中文本传到对话框的交互
- [ ] 实现 URL 粘贴自动 fetch + 进入双栏的流程

## 后续

什么情况下应该回来重新审视？

- 内测数据显示 < 30% 用户用过划词功能 → 说明左右联动不够自然
- > 50% 用户反馈"看不懂英文"→ 考虑给左栏加默认翻译选项
- 移动端用户占比 > 40% → 重新设计移动端布局（双栏在手机上塞不下）
- Chrome 插件版（v1.0+）上线后 → 评估是否要简化 Web 版
